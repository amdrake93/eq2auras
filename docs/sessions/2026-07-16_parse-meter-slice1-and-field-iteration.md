# Session chronicle: 2026-07-15 → 07-16 — the Parse Meter, brainstorm to field-hardened

*The story between the docs. What exists is in `docs/`; this is how it happened and why. Written as context was saved off ahead of the meter's slice 2.*

## The arc

The DPS meter — shelved since the go-public work — came off the shelf and went the whole distance in one session: **brainstorm → spec → plan (two review loops each) → inline TDD implementation → merge → three field fix-flow rounds → slice-2 scoping.** It started as SPEC Part III "Phase 2: The Parse Meter" and ended as a live, field-hardened meter on `dev-latest 0.1.114`, with slice 2 scoped and a clean tree.

The design fell out of a deliberate five-thread brainstorm, each thread resolved before the next:

1. **Data source.** ACT is the data layer, full stop — poll ACT's *computed* model (`CombatantData`/`EncounterData`), never a parallel swing accumulator. The realization that sharpened it: even the things that *sounded* like they needed our own accumulator don't — rolling DPS is sampling polled totals, drill-down iterates ACT's own retained swings, and overheal simply *doesn't exist* (EQ2 logs only effective healing). Nothing to re-derive.
2. **Segments** mirror ACT's encounter list (slice 1 = current only).
3. **Metric registry** — one flat registry, ACT's ExportVariables names, direct property selectors (not ACT's formatter callbacks — those get wiped by the parser plugin's re-init and return strings, not numbers).
4. **Window** — interactive, *not* click-through (the deliberate divergence: the menu is the product); a three-axis model (click-through baseline / locked / interactive-content); a fat dumb viewport.
5. **Convergence** — the load-bearing one (below).

A guild request surfaced mid-brainstorm — "let me put an arbitrary extra number on the row" — and became the **two-tier row** (primary drives bar+sort, `secondaries[]` are text). The **visual companion** (browser mockups) settled the two visual calls cleanly: row layout A (overlaid, no rank) and the `(duration) title — metric | total` header.

## Why-chains worth remembering

- **Convergence: extract *now*, and the burden of proof is on NOT sharing.** I leaned toward deferring the shared-rendering extraction (build the meter standalone, extract later). Alex overrode it, and was right: we have two *concrete* consumers (timer + meter), which is exactly when extraction is correct, not premature; deferring just calcifies the timer code and grows a parallel copy. The guardrail has **two brackets** — no speculative generality (ceiling) *and* no lazy divergence (floor); a "little difference" like the timer's spark gets **absorbed as row config**, not forked. So slice 1 extracted `OverlayWindowBase` + a configurable `BarRowVisual` and *re-seated the timer onto them* — accepting timer-regression risk, verified clean in the field. The "element/group model" roadmap arc stopped being a scheduled big-bang and became **emergent through the meter**.
- **Verify ACT internals from the decompile, never from memory.** Twice this mattered. In the plan review I asserted `GetStrongestEnemy()` was no-arg from the engine doc's member-list notation — the reviewer decompiled and caught that the only overload takes `(string)`. Later, when a groupmate's damage showed *no rows*, I did *not* guess — dispatched a decompile-verification agent that fetched ACTv3, decompiled, and returned the exact mechanism. The pattern earned its keep both times.
- **`GetAllies()` is you-relative — and so is ACT's own mini parse.** The groupmate-not-shown bug root-caused to `GetAllies()` seeding from `charName` and growing by shared-combat edges: a groupmate you haven't linked to is unreachable, so before you act the list is *empty*. There is **no clean friend/foe flag** — `GetCombatantType()`'s first line routes through `GetAllies()`, so it's circular. The decompile's honest finding: **the mini parse has the same limitation** (its `ShowOnlyAllies` filter has a `list2.Count > 0` escape hatch, so pre-engage it shows *everything including the mob*). Decision (Alex): **mirror the mini parse exactly** — show `Items.Values`, apply ACT's escape-hatch filter, drop `"Unknown"` — warts included. The mob shows pre-engage and self-heals the instant you act. Inventing a classifier would mean re-deriving what ACT deliberately owns.
- **Config surfaces diverge even when rendering converges.** Slice-2 scoping produced the boundary: the meter configures itself **in its own in-window right-click menu** (Details-style); the **ACT tab** is the *timer* config surface, heading toward a WeakAuras-style editor (the element/group arc); the **meter opts out of that arc entirely.** So meter↔timer share the *rendering* substrate but diverge on *config UX*. This came out of Alex's "starting to hate the tab" — putting meter config in the menu keeps the already-cluttered tab from getting worse and decouples "configure the meter" from "redesign the tab." The trigger insight: *"that data point is a knob"* — secondary data points aren't a feature that needs a knob, they *are* the meter's first knob, so slice 2 stands up the whole menu config surface at once (secondaries + the deferred row height/width).
- **Row-reorder animation is deliberately absent.** Alex briefly wanted animated row movement, then remembered Details' mechanism: the bar *widths* converge before the rank swap, so the discrete rebind is masked — adding sliding rows would fight it. Slot-keyed pool, animate width only.

## Process notes

- **The writer-run automated review loop ran across every artifact type and never needed the owner.** Spec (2 rounds), plan (2 rounds), a scroll delta (1 round), and three field fix-flows — all to closure, zero Question findings, zero deadlocks. The reviewer independently decompiled ACT to verify the "faithful to the mini parse" claim rather than trusting it. Fix-flow discipline held even for near-trivial fixes, and it kept catching real things: a `MeterWindow` ctor NRE that neither CI nor Core tests would trip, a duplicated color literal, spec overclaims ("1:1 with the mini parse" when only two of ACT's four skip clauses were reproduced).
- **Merge-to-dev-then-iterate cadence.** Each batch merged to `main` for a `dev-latest` build Alex field-tested, then fed back. The meter defaults **off** (opt-in), so every merge was low-risk regardless — the real exposure was timer-regression, which the shared-substrate extraction made a first-class merge-gate concern.
- Implementation was **inline** (executing-plans + strict Core TDD, 139→142 green), per the repo's watch-along convention — subagents were used for *review*, not for writing.

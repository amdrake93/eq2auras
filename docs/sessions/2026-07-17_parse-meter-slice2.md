# Session chronicle: 2026-07-17 — Parse Meter slice 2, brainstorm to field-verified

*The story between the docs. What exists is in `docs/`; this is how it happened and why. Written as context was saved off ahead of slice 2b (secondary data points).*

## The arc

Slice 2 — the meter's **configuration surface + multi-window** — went the whole distance in one long session: **brainstorm → one spec → five increments (each plan-reviewed to closure, then built with inline Core TDD + a CI checkpoint) → merge to beta → ~7 field fix-flows → field-verified end-to-end.** It shipped incrementally on `dev-latest` (through `0.1.131`) and ended fully validated live with a partner, on a clean tree (`main` only).

The brainstorm settled the whole design in a five-thread visual-companion session (mockups earned their keep again): config surface = **Details' split** (right-click = metric picker + lock + new/close; a **⚙ cog** opens a per-window dark **settings window** for the knobs); **all knobs are per-window** because **multi-window is a first-iteration need** (DPS + HPS side by side); windows are **edge-resizeable when unlocked**; **one opacity knob**; row width via resize (not a knob). Then the load-bearing move: the whole design was **too big for one spec/plan**, so it decomposed into **5 increments** — multi-window → settings+opacity → row height → font → edge-resize — each its own plan → review → implement → CI cycle, off **one shared spec**.

## Why-chains worth remembering

- **Spec describes the system; the backlog holds the phasing.** Alex's instinct mid-brainstorm: a single spec written "whole 2a now" risks becoming a project doc full of orders-of-operations and deferments. Resolution: SPEC Part III stays a **present-tense description of the designed system**; the increment ordering, cadence, and deferrals live in the **backlog** (and per-increment plans). This kept the spec clean across the whole slice.
- **Integration-branch ship model.** Alex: "won't merge *unimplemented* spec into main — that diverges main's spec from its code." So the whole slice built on one branch (`meter-slice2`); spec + code merged to `main` **together** only once all 5 increments were in. Field-test was to be batched to the end — but in practice we **merged to beta and iterated** (the meter's opt-in/off, stable is pre-meter, so beta exposure was safe), which is what surfaced the field bugs fast.
- **The review-cadence correction (the one process miss).** I read "move on to the next increment assuming CI is green" as license to skip the *plan review* too, and implemented increment 2 from an **unreviewed plan** (which carried a convergence decision — exactly what the reviewer should vet). Alex caught it: *"did we review loop the plan for this increment?"* The fix: "move on assuming CI green" deferred the **field-test**, not the **review**. Standing cadence set and saved to memory ([[per-increment-review-cadence]]): **write plan → review loop to closure → implement only after approval.** Every later increment followed it; the loops kept catching real defects (a clone dropping opacity, a `_root` forward-reference that wouldn't compile in isolation, doc-drift stragglers).
- **Convergence guardrail, applied without re-litigating.** Live opacity and live width both needed to reach state that lives in the *shared* `BarRowVisual` (the fill element, the row width). Rather than duplicate or rebuild, we exposed one-line setters (`FillOpacity`, `SetRowWidth`) on the shared primitive — the guardrail's **floor bracket** ("the component is reachable; burden of proof is on *not* sharing"), with the timer simply never calling them. The documented guardrail let these calls be made once and moved past, no re-debate.
- **Live-apply without rebuilds.** "Retain elements, animate properties" extended to the knobs: opacity mutates brush/element opacity; row height sets `RootBorder.Height` in place; font re-stamps via `ApplyFont`; width sets the element width (fill re-lerps next poll); visible-rows re-renders the slot pool. Only settings changes touch visuals, never per-tick.
- **"New window" refined three times — a field why-chain.** Brainstorm said **clone the source's config**. Field: a cloned *locked* window opens **on top of** its source and can't be dragged clear → changed to **default, inherit nothing**. Field again: that makes users re-set their look every new window → landed on **appearance travels, behavior/geometry doesn't** — the settings-window knobs (row height, font, opacity) are a personal preference and carry; metric, lock, size, and position are per-window and start fresh at an offset. The precise line is *which UI surface set it*: settings-window knobs inherit, drag/menu-set state doesn't.

## Process notes

- **Field iteration was the real design surface.** CI verifies compile; the plan review verifies plan-vs-tree; neither can prove *runtime WPF behavior*. Every field bug was exactly that class, invisible until the box: a right-click crash (`ItemContainerStyle` for `MenuItem` applied to the menu's `Separator`s), the opacity slider slamming min↔max (WPF `LargeChange` defaults to 1.0 on a 0.3–1.0 range), the right resize-grip floating in transparent window slack, the bottom-edge resize-up computing from the raw cap instead of the shown row count. The **merge-to-beta-then-fix-flow** loop (spec+code on a branch, reviewer available, merged on Alex's call) carried all of it in one sitting.
- **The automated review loop ran on every plan** (+ the spec, + inc-2's plan *and* code combined after the cadence slip), to closure — mostly 1–2 rounds, zero deadlocks. It independently re-derived against the tree each time and caught defects TDD/CI wouldn't.
- **Inline execution + strict Core TDD** (Core 142 → 163 green). Plugin tasks are transcribe-only on the Mac (no WPF build), gated by the CI compile checkpoint — which earned its keep, catching a `MeterWindowCallbacks` accessibility error (public ctor, internal param type).
- **Expected-not-a-bug, recorded:** beta → pre-meter stable (`0.1.104`) → beta **scrubs the windows list** — DCJS drops the unknown `windows` key when the old build re-saves `settings.json`; self-resolves once stable carries the meter. Logged so the reviewer/next-session doesn't re-raise it.

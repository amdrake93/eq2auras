# Session chronicle: 2026-07-14 → 2026-07-15 — going public, dual channels, and the field's first bite

*The story between the docs. What exists is in `docs/`; this is how it happened and why. Written as context was saved off ahead of un-shelving the DPS meter.*

## The arc

Picked up the thread the prior chronicle handed off: guildmates asking for access meant **go public**. It landed as a full arc — spec → plan → implementation → merge — plus the operational tail that "public" actually drags in (a license, a public-repo README, an install guide, scrubbed history), and then the first field install bit back.

The design settled on **two channels off one trunk**: `stable` (default) and `dev-latest` (beta, one `BetaChannel` checkbox), with the updater installing **by identity equality on the release name, never by version order**. Stable is not a rebuild — it's an in-the-moment **promotion** of a playtested dev build's *exact bytes* (`promote.yml`, `workflow_dispatch`, records the source SHA). The repo went **GPL-3.0**; CI stopped vendoring ACT's exe and started **fetching ACT's own latest public release** at build time; the README moved to **dynamic shields badges** (retiring the machine-stamped status block); a player-facing **README rewrite + `docs/install.md`** shipped. Work-account git identity was **scrubbed from all history** (filter-repo, force-pushed). First **stable `0.1.98`** cut, promotion pipeline verified.

Then the first real public install (via the new guide) reported the updater saying **"update available" forever**. Root-caused from the *actual shipped DLL bytes*, fixed, and re-promoted — closing at **both channels on `0.1.104`**, public/stable solid.

## Why-chains worth remembering

- **Public retired the entire token scheme — the simplification cascaded.** Public release assets download unauthenticated, so the fine-grained-PAT-in-DPAPI model was deleted wholesale: no token-entry UI, `TokenStore` gone, the self-updater fetches `browser_download_url` directly, and the self-hosted-runner escape hatch was foreclosed as moot. Going public *removed* code rather than adding it.
- **Identity equality, NEVER version order — and this is load-bearing, not stylistic.** Opting out of beta routinely moves *numerically backward* (`0.1.200` → `0.1.150`) and must still install. So the updater asks "is this a different build than what I'm running?", never "is this newer?". Toggling the channel checkbox currently triggers an immediate check against the now-selected channel (later reconsidered — see Standing at close).
- **Stable is a promotion, not a rebuild — "test what you ship."** A stable release republishes one dev build's exact bytes under the `stable` tag, never a recompile, so the bytes the public receives are the bytes that were playtested. `promote.yml` defaults to whatever `dev-latest` currently holds and records the source commit SHA; naming an older version is the rarely-used escape hatch.
- **CI tracks live ACT deliberately, not a pin.** A public repo can't redistribute ACT's exe, so CI downloads ACT's *latest* public release (`ACTv3.zip`) and builds against it. Latest-not-pinned is a choice: an ACT API break surfaces as a **build failure** rather than hiding until runtime. Behavioral breaks still need live/canary verification.
- **The field's first bite: the "identity" was mis-read, not mis-designed.** The updater compared the release name (bare `0.1.98`) against the assembly's `InformationalVersion` — which the **.NET SDK stamps with `+<sha>`** (`0.1.98+362dc…`; *not* SourceLink, no such package — the initial triage guessed wrong). Raw string inequality → never equal → nagged every launch, re-downloaded every check. The fix normalizes both sides to their **build-metadata-free semver core** (truncate at the first `+`) — per semver, `+build` is excluded from identity, and the `0.1.<run>` core (unique per CI run) is the true identity token. The design (equality-not-ordering) was right all along; the *identity string* had a hidden component. Lesson banked: **verify identity from the actual shipped bytes, not from what you assume the build emits** — the root cause came from reading the metadata out of the downloaded DLL, which is also why the prior triage's SourceLink mechanism got corrected.
- **Self-update writes one path, reloads another (found while writing the guide).** The updater writes new bytes to a **hardcoded** `%APPDATA%\…\Plugins`, but `ReloadSelf` re-reads the plugin's **registered** path (`PluginGetSelfData(this).pluginFile`). Install the DLL from anywhere else (e.g. Downloads) and the update writes bytes ACT never reloads — a **silent no-op**. Shipped the **docs workaround** (guide directs install into the Plugins folder so the paths coincide); the robust code fix (overwrite `pluginFile.FullName`) is backlogged as Option B, with an in-tree precedent at `ThirdParty/ACT_English_Parser.cs`.
